## Part 1: Minimize `packages/sdk/openapi.json` diff

### Problem

The `openapi.json` diff is ~5000 lines but **zero schemas were removed**. The diff is large because:

1. **Formatting changes**: All `"required"` / `"enum"` / `"tags"` multi-line arrays were collapsed to single-line (likely Prettier behavior changed — possibly a dependency that generates the base JSON before Prettier runs)
2. **Schema reordering**: Route order changed in `server.ts` (`/remote` moved earlier, `/network` added), causing schemas to appear in different positions
3. **`FileDiff` → inline**: The `Session.diffs` and `GlobalSession.diffs` fields changed from `$ref: FileDiff` to an inline object definition

The actual new content is small:

- 4 new schemas: `SessionNetworkWait`, 3 network event types
- 5 new API paths: 3 network endpoints, 2 KiloClaw endpoints
- 5 modified schemas: `Config` (+`remote_control`), `Event` (+ network refs), `GlobalSession`/`Session` (`diffs` inlined), `SessionStatus` (+`offline` variant)
- 4 modified paths: session-import endpoints (+`skipped` field)

Similarly, `types.gen.ts` shows types "removed" then "added" at a different position — they were just reordered.

### Approach: Regenerate with minimal route reordering

The openapi.json is auto-generated by `./script/generate.ts`. The formatting difference comes from running `prettier` on the generated output. To minimize the diff:

1. **Keep the `/remote` route in its original position in `server.ts`** — it was moved from after `/telemetry` to after `/global`, which reordered schemas in the generated output. Move it back to its original position.

2. **Regenerate `openapi.json`** via `./script/generate.ts`. The new endpoints/schemas will be added, but existing schemas will stay in the same order.

3. **If formatting diffs persist after regeneration** (due to Prettier), take the main branch's `openapi.json` and manually merge in only the new content:
   - New paths: `/network`, `/network/{requestID}/reply`, `/network/{requestID}/reject`, `/kilo/claw/status`, `/kilo/claw/chat-credentials`
   - New schemas: `SessionNetworkWait`, `Event.session.network.asked`, `Event.session.network.replied`, `Event.session.network.rejected`
   - Modified schemas: `SessionStatus` (+offline variant), `Config` (+remote_control), `Event` (+network refs), `Session`/`GlobalSession` (diffs field)
   - Modified paths: session-import endpoints (+skipped)

4. **Regenerate SDK types** from the (now minimal-diff) openapi.json. The types will preserve original ordering, so `types.gen.ts` and `sdk.gen.ts` diffs will be clean additions only.

## Part 2: Network Reconnection Notification

### Current flow

1. Network error detected in `session/processor.ts` → calls `SessionNetwork.ask()`
2. Session status set to `{ type: "offline", requestID, message }`
3. TUI shows `NetworkPrompt` with static text:
   - **"Network reconnect required"** (title)
   - `{error message}` (e.g., "Connection refused")
   - "Reconnect the network-dependent service, then press Enter to resume."
   - "Press Esc to stop this turn."
4. User manually presses Enter → session retries | Esc → session stops

### Problem

The user must manually detect when the network is back and press Enter. There's no automatic detection of network restoration, and no notification when connectivity returns.

### Desired flow

1. Network error detected → session pauses, TUI shows "waiting" state
2. System polls for connectivity restoration in the background
3. When network comes back → bell rings + prompt updates to ask: "Network resumed. Continue this turn?"
4. User presses Enter (resume) or Esc (stop)

### Implementation

#### 2a. Add network connectivity monitor (`session/network.ts`)

Add a `monitor()` function that polls for connectivity after a network wait is created:

```ts
async function monitor(info: Wait, abort: AbortSignal) {
  const interval = 3000 // poll every 3 seconds
  while (!abort.aborted) {
    await new Promise((r) => setTimeout(r, interval))
    if (abort.aborted) return
    const online = await probe().catch(() => false)
    if (online) {
      Bus.publish(Event.Restored, { sessionID: info.sessionID, requestID: info.id })
      return
    }
  }
}
```

The `probe()` function should make a lightweight connectivity check. Options:

- **DNS resolution** of a known host (e.g., `dns.google`)
- **HTTP HEAD** to a well-known endpoint
- **TCP connect** to a known port

Simplest and most reliable: DNS resolution via `Bun.dns.resolve("dns.google")` or similar.

#### 2b. New event: `SessionNetwork.Event.Restored`

Add a new bus event so the TUI can react when connectivity returns:

```ts
Restored: BusEvent.define(
  "session.network.restored",
  z.object({
    sessionID: z.string(),
    requestID: z.string(),
  }),
)
```

This needs to be:

- Published from the monitor when probe succeeds
- Included in the SSE event stream so the SDK receives it
- Added to the openapi.json schema
- Handled in `sync.tsx` → update the wait item to indicate restoration

#### 2c. Update `SessionNetworkWait` schema

Add a `restored` boolean field (default `false`) to track whether connectivity has returned:

## Part 3: Default Request Timeout (2 minutes)

### Problem

When the network is unavailable (e.g. laptop sleep/wake, Wi-Fi disconnect), it takes far too long for requests to fail. The fetch hangs until the OS TCP stack gives up (60-120+ seconds depending on OS), with no application-level timeout enforcing a bound. Three factors compound:

1. **No default fetch timeout applied**: `provider.ts` line 1143 checks `options["timeout"] !== undefined && !== null`, but when the user hasn't configured a timeout in their config, `options["timeout"]` is `undefined` and the entire timeout block is skipped. The config schema documents "Default is 300000 (5 minutes)" but **no code applies this default**.

2. **Bun's built-in timeout disabled**: Line 1174 passes `timeout: false` to Bun's fetch, explicitly disabling the runtime's own timeout.

3. **Unlimited retries by default**: `KILO_SESSION_RETRY_LIMIT` is `undefined` by default. The processor check at line 479 (`Flag.KILO_SESSION_RETRY_LIMIT === undefined || attempt < limit`) means retries are infinite. Each retry waits for another OS-level TCP timeout before failing.

Combined: each request can hang for minutes, then retries infinitely with backoff.

### Approach

Add a single constant `REQUEST_TIMEOUT_MS = 120_000` (2 minutes) and apply it as the default timeout in the provider's custom fetch wrapper. Also ensure timeout errors (`DOMException` with `name === "TimeoutError"`) are recognized as network disconnection, so they trigger the existing offline prompt flow.

### Changes

#### 3a. Add default timeout constant and apply it (`provider/provider.ts`)

Add a prominently placed, exported constant near the top of the file:

```ts
/** Default timeout (ms) for provider HTTP requests. */
export const REQUEST_TIMEOUT_MS = 120_000 // 2 minutes
```

Modify the custom fetch wrapper (lines 1143-1151) to apply the default when no explicit timeout is configured:

```ts
// Before:
if (options["timeout"] !== undefined && options["timeout"] !== null) {
  const signals: AbortSignal[] = []
  if (opts.signal) signals.push(opts.signal)
  if (options["timeout"] !== false) signals.push(AbortSignal.timeout(options["timeout"]))
  const combined = signals.length > 1 ? AbortSignal.any(signals) : signals[0]
  opts.signal = combined
}

// After:
const ms = options["timeout"] ?? REQUEST_TIMEOUT_MS
if (ms !== false) {
  const signals: AbortSignal[] = []
  if (opts.signal) signals.push(opts.signal)
  signals.push(AbortSignal.timeout(ms))
  opts.signal = signals.length > 1 ? AbortSignal.any(signals) : signals[0]
}
```

The key change: `options["timeout"] ?? REQUEST_TIMEOUT_MS` means:

- If user configured `timeout: 300000` → uses 300000
- If user configured `timeout: false` → disables timeout
- If not configured (`undefined`) → uses `REQUEST_TIMEOUT_MS` (120000)

#### 3b. Recognize timeout errors as network disconnection (`session/network.ts`)

`AbortSignal.timeout()` fires a `DOMException` with `name === "TimeoutError"`. The Vercel AI SDK may wrap this in its own error types, so we need to check the cause chain.

Update `disconnected()` to also match `TimeoutError` DOMExceptions:

```ts

```

Update `message()` to return a friendly message for timeout errors:

```ts
export function message(err: unknown) {
  // Check for timeout first (before code-based checks)
  for (const item of chain(err)) {
    if (item instanceof DOMException && item.name === "TimeoutError") return "Request timed out"

```

This ensures timeout errors enter the offline prompt flow where DNS polling will confirm whether the network is actually down. If the network IS up (provider was just slow), the DNS probe immediately succeeds and the user sees the restore prompt.

## Decisions (all parts)

- **Connectivity probe**: `Bun.dns.lookup("dns.google")` — fast, lightweight DNS check. Sufficient for sleep/wake scenarios.
- **Request timeout default**: 120,000ms (2 minutes) — generous enough for slow providers, short enough to detect offline quickly.
- **Timeout error classification**: Treat `TimeoutError` as network disconnection → enters offline prompt flow → DNS probe confirms actual network state. Worst case (provider is just slow): DNS probe immediately succeeds, user sees restore prompt, presses Enter.
- **Single constant**: `REQUEST_TIMEOUT_MS` in `provider/provider.ts` — one place to update for all providers. User can still override per-provider via config `timeout` field.
