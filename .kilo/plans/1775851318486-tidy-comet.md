# Re-implement kilocode changes on new `packages/opencode/src/permission/next.ts`

## Context

The upstream `next.ts` was refactored to use an Effect-based `PermissionService` (`service.ts`). The old kilocode changes were lost. The new `next.ts` is a thin adapter that delegates to `service.ts` via `runPromise`. We need to re-add all kilocode-specific functionality.

The old kilocode changes provided:
1. **`toConfig()`** — inverse of `fromConfig`, converts rules back to config format
2. **`saveAlwaysRules()`** — granular always-allow/deny rule persistence (used by `/permission/:requestID/always-rules` route)
3. **`allowEverything()`** — toggle allow-all mode per-session or globally (used by `/permission/allow-everything` route)
4. **`pending()`** — get a single pending request by ID
5. **Config protection** — force "ask" for config file edits, disable "always" for config files
6. **Session-scoped rules** — `s.session[sessionID]` local ruleset evaluated alongside approved rules
7. **Null sentinel filtering** in `fromConfig` — skip `null` entries from config patches
8. **`drainCovered()`** integration — auto-resolve sibling pending permissions after rule changes
9. **`ruleset` stored on `PendingEntry`** — needed by `drainCovered` and `saveAlwaysRules`
10. **Persist always-rules to global config** in `reply("always")`

## Architecture Decision

The new code splits into two layers:
- **`service.ts`** — Effect-based `PermissionService` with `ask`, `reply`, `list` (upstream code)
- **`next.ts`** — namespace adapter that wraps `service.ts` and adds non-Effect exports

**Strategy**: Add kilocode features in `next.ts` (the adapter layer) to minimize changes to shared `service.ts`. Where service-level changes are unavoidable (e.g., storing `ruleset` on `PendingEntry`, adding `session` to state, config protection in `ask`/`reply`), annotate them with `kilocode_change` markers in `service.ts`.

## Changes

### 1. `packages/opencode/src/permission/service.ts` — Minimal shared-code changes

These changes are necessary because the service owns the state and Effect logic:

#### 1a. Add `session` to `State` type
```ts
type State = {
  pending: Map<PermissionID, PendingEntry>
  approved: Ruleset
  session: Record<string, Ruleset> // kilocode_change
}
```
Initialize `session: {}` in the `InstanceState.make` factory.

#### 1b. Add `ruleset` to `PendingEntry`
```ts
interface PendingEntry {
  info: Request
  ruleset: Ruleset // kilocode_change
  deferred: Deferred.Deferred<void, RejectedError | CorrectedError>
}
```
Store `ruleset` when creating pending entries in `ask`.

#### 1c. Config protection in `ask`
- Import `ConfigProtection` from `@/kilocode/permission/config-paths`
- Evaluate with `local` session rules: `evaluate(permission, pattern, ruleset, state.approved, local)`
- After evaluating `allow`, check `ConfigProtection.isRequest(request)` and if true, override to `ask` behavior
- Inject `disableAlways` metadata key when config-protected

#### 1d. Config protection in `reply("always")`
- Import `ConfigProtection`
- Before persisting always-rules, check `ConfigProtection.isRequest(existing.info)` — if true, downgrade to `once` (resolve deferred, return early)
- When evaluating pending siblings after "always", include `item.ruleset` in `evaluate()` call
- After resolving pending siblings, persist always-rules to global config via `Config.updateGlobal`

#### 1e. Export new service methods on `PermissionService.Api`
Add to the interface:
- `saveAlwaysRules` 
- `allowEverything`
- `pending`

Implement them inside the `Layer.effect` block as Effect functions.

#### 1f. `DeniedError` constructor compatibility
The old `DeniedError` accepted a `Ruleset` directly. The new one uses `Schema.TaggedErrorClass` with `{ ruleset: Schema.Any }`. The `drain.ts` currently calls `new DeniedError(ruleset)` — need to update to `new DeniedError({ ruleset })` in `drain.ts`, or keep compatibility. Since `drain.ts` is kilocode-owned, update `drain.ts`.

### 2. `packages/opencode/src/permission/next.ts` — Adapter additions

#### 2a. Add `toConfig()` function
Port the `toConfig` implementation with `SCALAR_ONLY_PERMISSIONS` set. This is a pure function that doesn't need the service.

#### 2b. Add null sentinel filtering to `fromConfig()`
Add `if (value === null) continue` after the string check, and filter nulls from object entries.

#### 2c. Add `saveAlwaysRules` export
Delegate to `runPromise((service) => service.saveAlwaysRules(input))`.

#### 2d. Add `allowEverything` export  
Delegate to `runPromise((service) => service.allowEverything(input))`.

#### 2e. Add `pending` export
Delegate to `runPromise((service) => service.pending(input))`.

### 3. `packages/opencode/src/kilocode/permission/drain.ts` — Update for new types

The `drain.ts` file needs to be updated for the new architecture:
- The `PendingEntry` now uses `Deferred` instead of `resolve`/`reject` callbacks
- `DeniedError` is now an Effect `TaggedErrorClass` requiring `new DeniedError({ ruleset })`
- **Option A**: Rewrite `drainCovered` as an Effect function that receives the state and uses `Deferred.succeed`/`Deferred.fail`
- **Option B**: Keep `drainCovered` as a plain async function but adapt the interface

Since `drain.ts` is kilocode-only code, **Option A** is cleaner — rewrite it as an Effect function that operates directly on the `State` type from `service.ts`.

### 4. No changes needed

- `config-paths.ts` — unchanged, works as-is
- `routes.ts` — unchanged, already calls `PermissionNext.allowEverything`, `PermissionNext.toConfig`
- `server/routes/permission.ts` — unchanged, already calls `PermissionNext.saveAlwaysRules`
