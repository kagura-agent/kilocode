# Test Plan: Reproduce Issue #8391

## Issue Summary

When both `permission.external_directory` and `permission.read` are set to `"allow"` in config, Kilo still repeatedly prompts for approval on external-directory reads during task execution.

**Repro steps from issue:**

1. Set `permission.external_directory` to allow all paths
2. Set `permission.read` to allow all paths
3. Start a task that reads files outside the current workspace
4. Observe Kilo still asks for approval for external-directory reads

## Permission Flow for External Directory Reads

1. Read tool resolves absolute file path
2. `assertExternalDirectory()` checks if path is outside project:
   - If outside: calls `ctx.ask({ permission: "external_directory", patterns: ["/ext/dir/*"], always: ["/ext/dir/*"] })`
3. Read tool calls `ctx.ask({ permission: "read", patterns: [filepath], always: ["*"] })`
4. Each `ctx.ask` → `Permission.ask()`:
   - Merges rulesets: `agent.permission + session.permission + approved + local`
   - For each pattern, calls `evaluate(permission, pattern, ...rulesets)`
   - `evaluate` uses `findLast()` (last matching rule wins)
   - If action is "allow" AND `!isProtected` → continue (no prompt)
   - If action is "ask" OR isProtected → `needsAsk = true`

## Rule Ordering

Default agent ruleset includes `{ permission: "external_directory", pattern: "*", action: "ask" }`. User config rules are appended after defaults (via `Permission.merge(defaults, agentOverrides, user)`). Since `evaluate` uses `findLast`, user's `{ permission: "external_directory", pattern: "*", action: "allow" }` should win.

After all merges, `Truncate.GLOB` is appended (an `external_directory` allow for a specific path), which shouldn't interfere with general matches.

## Tests

### Test 1: `Permission.evaluate` returns "allow" for external_directory when user config overrides defaults

Unit test. Build a ruleset that mirrors the real merge order: defaults (with `external_directory: { "*": "ask" }`) followed by user config (`external_directory: "allow"`). Call `Permission.evaluate("external_directory", "/some/external/dir/*", merged)` and assert the action is `"allow"`.

Verifies the basic `findLast` semantics work for the `external_directory` permission.

### Test 2: `Permission.evaluate` returns "allow" for read when user config overrides defaults

Unit test. Build a merged ruleset where defaults have `read: { "*": "allow", "*.env": "ask" }` and user adds `read: "allow"`. Verify that `evaluate("read", "/some/external/dir/file.txt", merged)` returns "allow".

### Test 3: Full agent permission merge with config — external_directory "allow" sticks

Integration test with `tmpdir({ config: { permission: { external_directory: "allow", read: "allow" } } })`. Load `Config.get()`, build the agent permission array using the same logic as `agent.ts` (merge `defaults + user`), then verify `Permission.evaluate` returns "allow" for external directory patterns.

### Test 4: `Permission.ask` returns immediately without blocking when external_directory is "allow"

Integration test using `Permission.ask()` directly. Provide a ruleset with `external_directory: "allow"` (`{ permission: "external_directory", pattern: "*", action: "allow" }`). Call `Permission.ask({ permission: "external_directory", patterns: ["/ext/dir/*"], ... })`. Assert it resolves immediately (doesn't block waiting for user reply).

### Test 5: `Permission.ask` returns immediately for read when "allow" is configured

Same as Test 4 but for `read` permission with pattern matching an external file path.

### Test 6: Read tool does not ask for external_directory when agent ruleset has external_directory "allow"

Effect-based test (extend existing `read.test.ts` pattern). Use the `asks()` helper to capture permission requests. Build a `ctx` whose `ask` evaluates rules like the real `Permission.ask` does — checking the agent's ruleset. Set up an agent ruleset where user config has `external_directory: "allow"`. Read an external file. Assert that `ctx.ask` is called but the `external_directory` request evaluates to "allow" (no blocking).

### Test 7: Read tool does not prompt for both external_directory AND read when both are "allow"

Same setup as Test 6 but verify NEITHER `external_directory` nor `read` triggers a prompt. Read an external file. Count how many permission requests would need user interaction (based on rule evaluation). Assert zero.

### Test 8: Multiple different external directories do not prompt when external_directory is globally "allow"

Read files from 3 different external directories. Verify none trigger a prompt. This tests the repeated-prompt scenario from the issue — ensuring a global "allow" covers all directories, not just previously approved ones.

### Test 9: "always" approval for one directory still prompts for different directories (default config)

Negative test. With DEFAULT config (external_directory: ask), approve "always" for `/ext/dir1/*`. Then request `/ext/dir2/*`. This SHOULD still prompt because "always" rules are directory-scoped. This verifies the distinction between "always for one dir" vs config-level "allow all".

### Test 10: Truncate.GLOB appended after user config does not override external_directory "allow"

Unit test. Simulate the agent.ts post-processing that appends `Truncate.GLOB` allow rule. Verify that for an arbitrary external directory path, the evaluation still returns "allow" from the user's wildcard rule, not "ask" from defaults.

### Test 11: ConfigProtection.isRequest returns false for read tool's external_directory requests

Unit test. Verify that `ConfigProtection.isRequest({ permission: "external_directory", patterns: [...], metadata: { filepath: "/some/path" } })` returns `false`. The `metadata.filepath` presence means it's a file-tool request, which should bypass config protection.
